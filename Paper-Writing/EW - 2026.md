## 1. The Concept: Data Mesh at the U-space Edge

Data Mesh is not just a storage pattern; it is a socio-technical architecture based on four pillars. When applied to U-space edge deployment (using 5G/6G MEC - Multi-access Edge Computing), these pillars transform how tactical data is handled:

### The Four Pillars in U-space

|**Data Mesh Pillar**|**U-space Edge Realization**|
|---|---|
|**Domain Ownership**|Data is owned by local "cells" (e.g., a port, a city district, or a 5G subnet). The local **USSP (U-space Service Provider)** at the edge manages its own tactical telemetry.|
|**Data as a Product**|Instead of raw bitstreams, the edge node provides "Tactical State Products" (cleaned, timestamped, and validated trajectories) via standardized APIs for other drones or providers.|
|**Self-Serve Platform**|5G/6G network slicing acts as the infrastructure. A drone "plugs in" to a slice and immediately gains access to the local data mesh without custom integration.|
|**Federated Governance**|Global ICAO/EASA rules (e.g., separation minima) are encoded as "Computational Policies" that run automatically at the edge to ensure cross-domain interoperability.|

## 2. Realizing Tactical Separation via Edge Mesh

Under EASA Regulation 2021/664, tactical separation is the "reflex" action to prevent collisions.

- **The 5G/6G Link:** Using **URLLC (Ultra-Reliable Low-Latency Communications)**, the edge node acts as a **Distributed Tactical Coordinator**.
    
- **The Data Mesh Flow:** 1. Drone A and Drone B belong to different USSPs but fly in the same 5G cell.
    
    2. Instead of sending data to a central Cloud CIS (Common Information Service), the edge node creates a local **Data Product** of the immediate airspace.
    
    3. The **Tactical Separation Service** runs locally on the MEC, consuming these products to issue "Resolution Advisories" (maneuver commands) in $<10ms$.

For a conference paper, the following benefits are high-value:

## 4. Gaps and Research Challenges

To make this a "State-of-the-Art" paper, you must address these unsolved gaps:

- **Mesh Handover (Mobility):** How does a "Data Product" follow a drone moving at 100 km/h across 5G cell boundaries? The state-of-the-art lacks a protocol for **Stateful Edge Handoff** in a mesh.
    
- **Trust & Verification:** In a mesh, you consume data from other "domains." If an edge node is compromised, it could feed "Ghost Drones" into the tactical service. 6G **Zero-Trust** architectures are needed here.
    
- **Semantic Interoperability:** ICAO provides high-level frameworks, but there is no "Data Contract" standard (like JSON Schema or Protobuf definitions) specifically for U-space tactical mesh products.
    
- **Regulatory Alignment:** EASA currently expects a degree of centralization (the **CISP** - Common Information Service Provider). A pure Data Mesh challenges the "Single Source of Truth" legal requirement by introducing "Distributed Truth."

## 1. System Model: The "6G-UTM Data Mesh"

We consider a 6G-based U-space environment composed of three hierarchical layers that operate as a **Data Mesh**:

1. **The Aerial Domain (Drones/UEs):** High-density UAVs acting as mobile mesh nodes. Each drone is a "Data Producer," generating high-frequency $(\geq 50 \text{ Hz})$ telemetry and intent data.
    
2. **The Edge Domain (6G gNB/MEC):** The 6G base stations (gNB) with integrated **Multi-access Edge Computing (MEC)** act as "Domain Owners." Each MEC node manages a specific U-space volume. Unlike 5G, the 6G gNB performs **ISAC (Integrated Sensing and Communication)** to verify drone positions independently of telemetry.
    
3. **The Data Mesh Plane:** A decentralized logical layer where "Tactical State Products" are exchanged. Instead of a central Cloud database, the MEC nodes use a **Distributed Ledger** or **Discovery and Synchronization Service (DSS)** to share real-time conflict buffers with neighboring cells.
    

$$S_{total} = \sum_{i=1}^{N} Domain_i(MEC_i, \{UAV_{i,j}\})$$

Where each $Domain_i$ provides a **Tactical Separation Service** locally, ensuring that the latency $\tau$ remains within the xURLLC (extreme URLLC) threshold of $\tau < 1 \text{ ms}$.

## 2. Problem Statement

The transition from EASA U2 (initial) to **U3 (advanced)** services requires **Tactical Conflict Resolution (TCR)** in sub-second intervals. Conventional centralized UTM architectures fail due to:

- **Backhaul Bottlenecks:** Centralized cloud processing introduces "tromboning" delays (UAV $\rightarrow$ Edge $\rightarrow$ Core $\rightarrow$ Cloud $\rightarrow$ Core $\rightarrow$ Edge $\rightarrow$ UAV), exceeding the safety-critical window for mid-air collision avoidance.
    
- **Data Silos:** In a multi-provider (Multi-USSP) environment, sharing raw telemetry is inefficient and raises privacy concerns.
    
- **Dynamic Topology:** The 6G Air-to-Ground (A2G) link is highly volatile. At high drone densities, the probability of packet loss $P_{loss}$ increases, directly impacting the reliability of the separation minima.
    

**Core Problem:** How can we orchestrate a decentralized Data Mesh at the 6G edge to guarantee **xURLLC-grade** tactical separation while maintaining data sovereignty and seamless stateful handover between airspace domains?

## 3. Research Questions (RQs)

These questions are designed to trigger technical discussion in a 6G/Wireless paper:

- **RQ1 (Latency & Reliability):** Can a **Cell-Free Massive MIMO** 6G architecture, combined with Data Mesh principles, achieve the $10^{-7}$ reliability required for U3 tactical separation in "Deep Urban" canyons where Multipath Interference is high?
    
- **RQ2 (Stateful Handover):** How can we design a **Context-Aware Handover** protocol that transfers the "Tactical State Product" (computational state) of a UAV between edge mesh nodes faster than the 6G radio handover itself?
    
- **RQ3 (Semantic Communication):** Instead of transmitting raw coordinates, can we use **Semantic 6G Communication** to transmit only "Conflict Intent," thereby reducing the required bandwidth for the Data Mesh by an order of magnitude?
    
- **RQ4 (Distributed Trust):** In a decentralized U-space, how do we implement a **Zero-Trust Physical Layer Authentication** to prevent malicious actors from injecting "Ghost Drones" into the local edge mesh?

### 2. The Data Mesh Pipeline: Subscribe & Flow

In a Data Mesh, we replace "API Pulling" with **Event-Driven Streams**.

#### Step-by-Step Data Flow:

1. **Ingestion (The Sidecar):** The UAV sends telemetry via 6G URLLC to the gNB.
    
2. **Domain Translation:** The MEC "Sidecar" (a lightweight proxy) converts the raw 6G packets into a standardized **Tactical Data Product** (e.g., a Protobuf object containing 4D coordinates and velocity).
    
3. **The Mesh Bus:** This product is published to a local **Edge Data Bus** (e.g., Zenoh or MQTT+).
    
4. **Cross-USSP Consumption:** * _USSP-A’s_ Conflict Detection service "subscribes" to all products in the local cell.
    
    - It processes the telemetry of its own drones **and** competitors' drones (as anonymized data products) to run the **Tactical Separation** algorithm.

### 3. Simulation Metrics: Latency & Handover

For your simulation, these are the primary variables to measure the efficiency of the Data Mesh vs. a Centralized Cloud.

#### A. Latency Components ($\sum \tau$)

In a Data Mesh, the goal is to keep the "Age of Information" (AoI) below the tactical threshold (typically $50 \text{ ms}$ for U3 services).

- **$\tau_{air}$:** Air-interface latency (6G URLLC target: $< 1 \text{ ms}$).
    
- **$\tau_{mesh}$:** The time to transform raw telemetry into a "Data Product" and distribute it to other USSP containers on the MEC (Target: $< 2 \text{ ms}$).
    
- **$\tau_{proc}$:** Time for the USSP tactical algorithm to calculate a deconfliction maneuver.
    

#### B. Handover Metrics (The "Mesh Handoff")

When a UAV moves from **Cell 1 (MEC A)** to **Cell 2 (MEC B)**, the simulation must track:

- **State Transfer Latency:** How long it takes for the "Data Product" (the drone's digital twin) to migrate from MEC A’s bus to MEC B’s bus.
    
- **Inter-MEC Synchronization:** If the UAV is on the border, both MECs must "mesh" their data. The metric is the **Jitter in Coordination**, ensuring both USSPs see the same drone position simultaneously.
